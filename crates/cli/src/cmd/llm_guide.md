# pyllow for AI agents

You are about to use **pyllow** — a static analysis CLI for Python codebases.
This document is your operating manual. Read all of it before acting on any
output. Pyllow surfaces _candidates_; you must verify before applying changes.

## What pyllow analyzes

| Subcommand | What it finds |
|---|---|
| `pyllow check [path]` | Unused files / imports / dependencies, circular imports |
| `pyllow dupes [path]` | Copy-paste duplication (4 modes — see below) |
| `pyllow health [path]` | Cyclomatic / cognitive complexity, maintainability index, hotspots, ranked refactor targets |
| `pyllow smells [path]` | Python anti-patterns (10 rules: mutable default, broad except, sentinel equality, etc.) |
| `pyllow flags [path]` | Feature-flag inventory (env vars, Django settings, LaunchDarkly / Statsig / Unleash / GrowthBook SDK calls) |
| `pyllow audit . --base <ref>` | PR-scoped run of all the above with a PASS / WARN / FAIL verdict |

All commands accept `--format {human,json,sarif,markdown}`. **Always use
`--format json` when parsing programmatically**; the human format includes
ANSI escape sequences and is laid out for terminals.

## How to read JSON output

Each issue has at minimum:

- `type` — kebab-case rule key (`unused-file`, `circular-dependency`, `mutable-default`, …)
- `path` — file relative to invocation root
- `line` — present for line-scoped issues, absent for file-scoped (e.g. `unused-file`, `low-maintainability`)
- additional fields specific to the rule (e.g. `cyclomatic` / `cognitive` for complexity, `cycle: [path, …]` for circular-dependency, `flag` / `provider` for feature-flag)

`pyllow check` runs the full pipeline; individual subcommands narrow the focus.

## Modes you must understand

`pyllow dupes --mode {strict|mild|weak|semantic}`:

- **strict** — exact tokens. Lowest false-positive rate, misses everything LLMs typically produce.
- **mild** _(default)_ — whitespace-normalized. Good general signal.
- **weak** — drops literal contents. Catches "different log message, same control flow."
- **semantic** — also drops identifier names. Designed to catch LLM rename-paste, but **aggressive on framework codebases**: FastAPI route signatures, ORM filter chains, pytest fixture patterns will look identical structurally without being real duplication. Pair with `--skip-local` and consider raising `--window` above the default 50.

## False-positive classes you MUST sanity-check before acting

These are framework-agnostic patterns where pyllow systematically over-reports:

1. **Generated / scaffolded code** — autogenerated `__init__.py` re-exports, ORM migrations, GraphQL schema files, OpenAPI clients. These are intentionally repetitive. Verify with `git log --oneline <path>` — short history with "generated" / "scaffold" in messages is a strong signal.

2. **Framework convention dupes** — handler signatures across web routes, model definitions across modules, fixture declarations across test files. Real duplication has the same _intent_; framework boilerplate has the same _shape_ but different intent. Read both occurrences side-by-side before "deduplicating."

3. **ORM `==` comparisons** — `Model.attribute == None` is required by SQLAlchemy / Beanie / Django ORM (they overload `==` to build query predicates). The smell rule `sentinel-equality` already skips class-attribute LHS, but check anyway if you see one flagged.

4. **`raise … from None`** — idiomatic in FastAPI for `HTTPException` to suppress trace leaking. The `raise-from-none` smell rule is correct as-is, but on FastAPI codebases consider disabling via `pyllow.toml`'s `[smells].disabled = ["raise-from-none"]`.

5. **Cyclic imports broken with local imports** — pyllow detects cycles in the import graph regardless of whether one side uses a function-local `import` to break the runtime cycle. The cycle still exists structurally; treat the finding as "this is fragile" rather than "this is broken."

6. **Long test files / fixture modules** — pyllow auto-skips test files in `dupes`, `low-maintainability`, and parts of `smells`. If a test file is still flagged, it's likely under a non-standard test directory (not `tests/` or `test/`).

7. **High complexity in pattern-matching code** — state machines, parsers, command dispatchers can have inherent CC/cog scores that should NOT be reduced. Surface them as "review" not "refactor."

## Verification recipes (do this before changing code)

For every category, the rule is the same: **read the actual code, do not trust the finding alone**.

| Finding type | How to verify |
|---|---|
| `unused-file` | `grep -rn "from <module>\|import <module>" --include="*.py"`. Also check `pyproject.toml` for `[project.scripts]` entry points, `__all__` re-exports, and dynamic loaders (`importlib.import_module`). |
| `unused-import` | Read the file. Check type-annotation strings, docstrings, `__all__`, and `if TYPE_CHECKING:` blocks. Run `python -c "import <module>"` or `ruff check --select F401 <file>` for cross-validation. |
| `unused-dep` | `grep -rn "<package_name>" --include="*.py"` against the package's actual import name (often differs from the dist name — e.g. `pillow` is imported as `PIL`). Check `pyproject.toml` `[project.optional-dependencies]` — extras may be conditionally used. |
| `circular-dependency` | Trace each path's `from X import …` chain. Common workaround is a function-local import on one side; the cycle is still structural. |
| `duplicate` | Open both `path:line-line` ranges side by side. Confirm intent matches, not just shape. Framework boilerplate looks duplicate but isn't. |
| `complexity` (high cc/cog) | Inspect control flow. State machines and parsers can legitimately have high scores. If the function does one thing in a complicated way, refactor; if it does many similar things, leave it. |
| `low-maintainability` | The MI formula penalizes high LOC + many tokens. Check if the file is a "data" file (constants, schemas, enums) — those score low without being a problem. |
| `hotspot` | High score = high complexity × git churn. Real signal, but verify the churn is actual logic changes vs. mechanical edits (rename, format). |
| smells (all rules) | Each rule has a precise definition (see the README rule-IDs). Read the surrounding code; many are intentional in specific contexts. |
| `feature-flag` | The flag name and provider are detected from the SDK call shape. Cross-reference with your flag-management system to confirm — pyllow has no view into runtime values or rollout state. |

## Suppression mechanisms (use sparingly, after verification)

Pyllow honors existing Python lint conventions — it does NOT introduce its own `# pyllow:`-prefixed dialect.

```python
# noqa                      → suppresses ALL pyllow rules on this line
# noqa: B006                → mutable-default
# noqa: BLE001              → broad-except
# noqa: E711, E712          → sentinel-equality
# noqa: T201                → stray-print
# noqa: F401                → unused-import
# noqa: S110, S112          → broad-except (security variant)

# ruff: noqa                → file-level suppress all
# flake8: noqa: T201        → file-level suppress only T201
```

Other suppression options:

- Add path globs to `.pyllowignore` (one per line, `#` for comments).
- Disable smell rules in `pyllow.toml` under `[smells].disabled = ["rule-key"]`.
- Use `--baseline FILE` to filter previously-known issues; combine with `--save-baseline FILE` once on a clean ref.
- Use `--save-snapshot` and `--trend` to track per-rule deltas across runs.

## Recipes

```bash
# First-look on an unfamiliar codebase
pyllow check . --format json | jq '.issues | length'
pyllow health . --top 10 --format markdown

# Find probable LLM rename-paste
pyllow dupes . --mode semantic --skip-local --format json

# Refactor candidates ranked by quick wins
pyllow health . --targets --effort low --format markdown

# PR-scoped quality gate (CI)
pyllow audit . --base main --format sarif > pyllow.sarif

# Establish a baseline so audit fails only on NEW issues
pyllow check . --save-baseline pyllow-baselines/check.json
pyllow audit . --base main --baseline pyllow-baselines/check.json
```

## Operating discipline

When a user asks you to "use pyllow output to fix X":

1. **Run the appropriate subcommand with `--format json`**, parse, count by `type`.
2. **Sample at least 3 instances per finding type** before deciding the type is real or noisy.
3. **Read the actual code** at each `path:line` you intend to act on.
4. **Apply the suppression mechanism, not a patch**, when the finding is intentional (the user knows their codebase better than pyllow does).
5. **Never apply mass changes** based on `pyllow fix`'s `--yes` mode without first running `--dry-run` and reviewing the diff.
6. **Report findings, don't act unilaterally** when complexity or maintainability findings exceed ~20 — the cost of a wrong refactor is much higher than the cost of asking.

## What pyllow is NOT

- Not a type checker (use `mypy` / `pyright`).
- Not a linter for stylistic issues (use `ruff`).
- Not a runtime profiler (no execution data, all static).
- Not aware of dynamic patterns it can't see: `__import__("dotted.path")`, `getattr(obj, "method_name")`, decorator-driven registration without explicit imports. Those will produce false positives in `unused-*` analyses.

If `pyllow check` flags something dynamic and you're confident it's used, add the entry point to `pyllow.toml`'s `entryPoints` field and re-run.
